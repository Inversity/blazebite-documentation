## Red Flags to Avoid

If your documentation has ANY of these, fix before publishing:

‚ùå **User must "figure out" any step** - every action must be explicit

‚ùå **Assumes knowledge not taught earlier** - be fully self-contained or link clearly

‚ùå **Paragraph over 3 sentences** - break it up immediately

‚ùå **Vague troubleshooting** - "try again" or "contact support" without diagnostics

‚ùå **Missing validation checks** - user can't tell if step succeeded

‚ùå **Technical jargon unexplained** - define or avoid

‚ùå **Generic placeholders still present** - all `[brackets]` must be replaced

‚ùå **Comments still in document** - all `<!-- -->` must be deleted

‚ùå **"Should see" without specifics** - say EXACTLY what appears

‚ùå **No time estimates** - users need to plan their setup

‚ùå **Support info missing** - users must know how to get help

‚ùå **Prerequisites untestable** - user can't verify they're ready

## Testing Your Documentation

Before finalizing:

1. **Dogfood it yourself**
   - Follow the guide step-by-step on actual hardware
   - Time yourself to verify time estimates
   - Note any confusion points

2. **Test with unfamiliar user**
   - Find someone who hasn't seen this device
   - Watch them follow your guide (don't help!)
   - Note where they struggle or ask questions

3. **Technical validation**
   - Verify all internal links work (especially cross-folder)
   - Check images display correctly
   - Confirm markdown renders properly
   - Test on mobile (many users read docs on phones)

4. **Run through quality checklist**
   - Complete entire checklist above
   - Fix any items that don't pass

5. **Final read-through**
   - Read aloud to catch awkward phrasing
   - Check for consistent terminology
   - Verify tone is helpful, not condescending

## Distinctions from Other Templates

### vs. TEMPLATE_FEATURE_REFERENCE.md
- **Onboarding**: Initial setup, first-time configuration, getting started
- **Reference**: How features work, what buttons do, detailed functionality
- **Rule**: If user needs it on Day 1 to get running ‚Üí Onboarding. If it's about using features after setup ‚Üí Reference.

### vs. TEMPLATE_TROUBLESHOOTING.md
- **Onboarding**: Inline troubleshooting for setup steps, quick fixes
- **Troubleshooting**: Detailed diagnostic procedures, complex problems, symptom ‚Üí solution mapping
- **Rule**: Setup-specific issues ‚Üí inline in Onboarding. Ongoing operational issues ‚Üí dedicated Troubleshooting doc.

### vs. Operational Procedures
- **Onboarding**: One-time setup and initial configuration
- **Operational**: Daily workflows, recurring tasks, shift procedures
- **Rule**: First-time only ‚Üí Onboarding. Every day/shift ‚Üí Operational doc.
